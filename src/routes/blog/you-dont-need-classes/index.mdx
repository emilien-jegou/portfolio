Classes have been a part of programming since the early days of JavaScript. Initially, developers used functions to effectively represent classes, which was sufficient for the needs of JavaScript at the time. However, as the language evolved, developers began to ask for more features.

The 2000s saw a significant influence of object-oriented programming, languages like PHP, C#, C++, and Java consolidated their presence, it was in this context that classes specific utilities were first introduced to javascript in 2009 through the ES6 (ECMAScript 6) proposal which was finalized as a standard in 2015.

The introduction of the 'class' keyword in javascript was largely driven by a desire for familiarity and structure. Although the addition of OOP specific keywords was not a technical necesitty, their introduction catered to developers accustomed to traditional OOP methods and provided a platform for "uniformity" accross differents programming languages.

Since their "shinier" counterpart arrived, function classes have slowly faded away, but should we really give them up entirely? This article is an opinionated excercise to provide a comparison between popular object-oriented patterns and libraries, and demonstrate what a class-less approach would look like in practice — enjoy!

## "Your code doesn't allow X"

A common criticism of class-based programming is its tendency toward over-engineering. Converting class-heavy code to more functional approaches frequently results in significant code reduction.

Developers who over-engineer solutions often do so anticipating future
extensions or requirements, in OOP this tendency is prevalent and can only be
explained by the highly codified practices practices developer have to follow
(`SOLID`, `DRY`, design patterns and so on). Yet a more efficient approach
would be implementing these hypothetical extensions only when genuinely
required, through focused refactoring. This just-in-time development strategy
keeps codebases cleaner, simpler, and more maintainable rather than cluttering
them with speculative functionality e.g.:
- The benefit of the Open-closed principle is minimal on the code you control.
- Dependency Inversion assume you define abstraction for code that may not need them.

Those principles may have made sense when they were first introduced in the
early 2000s but with the advent of LSP and types systems refactoring have never
been made easier, in 2004 languages made extra effort to not have a type-system
(think python, ruby, PHP) in that context any security guarantees in your code
were extrapolated to extreme importance.

## Inheritance, mixed results

The "composition over inheritance" principle seems widely accepted today, yet
there may be cases where working with multiple levels of inheritance is the
best option available, for instance (and unironically), if you are working on
top of legacy systems modelled using inheritance. A notable example is the
HTML/XML tree structure, where a hierarchical organization is inherent to the
underlying data.

The Servo JavaScript engine, built entirely in Rust, encountered this
challenge of modelling the html tree structure. As Rust does not support
inheritance, the developers had to rely on complex macros to mitigate the
overhead of modifying, resulting in significant code duplication ([see how
methods and properties are copied over in the HtmlAnchorElement](https://github.com/servo/servo/blob/03e953e22c5cb8ce3d7d375f6f9e841da46ba6f3/components/script/dom/htmlanchorelement.rs#L40)).

Similarly, the Ladybird browser and JavaScript engine initially considered
switching from C++ to Rust but eventually abandoned the idea due to the
inheritance barrier, instead opting for Swift, an object oriented language.

Luckily for you typescript is so flexible you may use inheritance without
resorting to class tokens, not that I'd expect you to develop a javascript
engine in javascript.

For our experiment we will build a simple html inheritance chain, both code
will use the following function for generating html:

```typescript
class Elem {
  public childrens: Elem[] = [];
  public attributes: Record<string, string> = {};

  constructor(public tagName: string) {}

  getHTML(): string {
    const childHtml = this.childrens.map((c) => c.getHTML()).join('');
```

## Opening up interfaces

So here we reach the point where you need to expose your code to the external
world, how would you build up an interface that would allow users to extend it,
without compromising on what you built so far?


<SideBySideCode language="typescript">
```typescript
abstract class Shape {
  abstract calcRadius(): number;
}

class Circle extends Shape {
  constructor(public radius: number) {
    super();
  }

  calcRadius(): number {
    return this.radius;
  }
}

class Square extends Shape {
  constructor(public sideLength: number) {
    super();
  }

  calcRadius(): number {
    return Math.sqrt((this.sideLength * this.sideLength) / Math.PI);
  }
}

const circle = new Circle(5);
const square = new Square(8);

console.log(`circle radius: ${circle.calcRadius()}`);
console.log(`square radius: ${square.calcRadius()}`);
```
```typescript
import { exhaustive } from "exhaustive";

type Circle = { kind: 'circle'; radius: number; }
type Square = { kind: 'square'; sideLength: number; }

// This type will fill the role of the abstract interface
// it will only be used externally (e.g. as exported from a library),
type AbstractShape = { kind: 'abstract'; calcRadius(): number; }

type Shape = Circle | Square | AbstractShape;

const createCircle = (radius: number): Circle => ({ kind: 'circle', radius, });
const createSquare = (sideLength: number): Square => ({ kind: 'square', sideLength, });

const calcRadius = (shape: Shape) => exhaustive.tag(shape, 'kind', {
  square: (s) => Math.sqrt((s.sideLength * s.sideLength) / Math.PI),
  circle: (c) => c.radius,
  abstract: (shape) => shape.calcRadius(),
});

// Example usage:
const circle = createCircle(5);
const square = createSquare(8);

console.log(`circle radius: ${calcRadius(circle)}`);
console.log(`square radius: ${calcRadius(square)}`);

const customShape = { kind: 'abstract', calcRadius: () => 3 };
console.log(`custom shape radius: ${calcRadius(customShape)}`);

// If the code above was "locked" for modification, you would have many way to
// extend it, one non-trivial example would be extending the Shape class.
// In OOP you would have to wrap and uses adapters to achieve this, in FP
// you could do so through dymamic interfaces:
type Shape<C extends AbstractShape = AbstractShape> = Circle | Square | C;

// ...

type CustomShape = AbstractShape & { calcArea(): number; };

const calcArea = (shape: Shape<CustomShape>) => exhaustive.tag(shape, 'kind', {
  square: (s) => s.sideLength * s.sideLength,
  circle: (c) => Math.PI * c.radius ** 2,
  abstract: (shape) => shape.calcArea(),
});
```

## Dependency Inversion

There are many packages for dependency inversion in typescript, a
prominent one is inversify, it uses similar mechanism as nest.js, although
the latter can be quite "hacky" in the way it resolve dependencies.

In both those crate the main advantage is that both the declaration of a
dependency and it's injection is done in the same place, the problem with
those crate is that you inevitably sacrifice type safety as the
dependencies are "assumed" to be loaded instead of ensured through the
typesystem.

In this example I will showcase how to achieve the same
behavior, without packages, and with proper typing through functional
typescript. the point is to prove that getting away from standard OOP will
bring  improvment with minimal effort.

<SideBySideCode language="typescript">
```typescript















































type WeaponKind = 'sword' | 'firearm';

type Weapon = {
  damage: number;
  kind: WeaponKind;
};

const ninjaWeaponId: symbol = Symbol.for('NinjaWeaponId');

@injectable()
class Katana implements Weapon {
  public readonly damage: number = 10;
  public readonly kind: WeaponKind = 'sword';
}

@injectable()
class Ninja {
  constructor(
    @inject(ninjaWeaponId)
    public readonly weapon: Weapon,
  ) {}

  // Perform a slash attack and returns it's damage
  slash(): number {
    switch (this.weapon.kind) {
      case 'sword':
        return 1.1 * this.weapon.damage;
      case 'firearm':
        return 0;
    }
  }
}

const container: Container = new Container();

container.bind(Ninja).toSelf();
container.bind(ninjaWeaponId).to(Katana);

const ninja: Ninja = container.get(Ninja);

console.log('Damage dealt:', ninja.slash());
```
```typescript
// Some utilities are needed to make this work
type Lazy<T> = () => T;

const lazy = <T>(initializer: () => T): Lazy<T> => {
  let initialized = false;
  let value: T | undefined;
  return (() => {
    if (!initialized) {
      value = initializer();
      initialized = true;
    }
    return value!;
  }) as Lazy<T>;
};

type Injected<K extends string, T> = Record<K, Lazy<T>>;

type ContainerBuilder<T extends Record<string, any>> = {
  bind: <K extends string, V>(key: K, value: V) => ContainerBuilder<T & Injected<K, V>>;
  lazy: <K extends string, V>(
    key: K,
    initializer: (context: T) => V,
  ) => ContainerBuilder<T & Injected<K, V>>;
  build: () => T;
};

const createContainer = <T extends Record<string, any>>(): ContainerBuilder<T> => {
  const context: Partial<T> = {};

  const builder: ContainerBuilder<T> = {
    bind<K extends string, V>(key: K, value: V) {
      (context[key] as any) = () => value;
      return this as any;
    },
    lazy<K extends string, V>(key: K, initializer: (context: T) => V) {
      (context[key] as any) = lazy(() => initializer(context as T));
      return this as any;
    },
    build() {
      return context as T;
    },
  };

  return builder;
};

// We will take a different approach in functional programming
type WeaponKind = 'sword' | 'firearm';

type Weapon = {
  damage: number;
  kind: WeaponKind;
};



const Katana: Weapon = {
  damage: 10,
  kind: 'sword',
};

// If you need multiple injection you would do it as such:
// Injected<'a', A> & Injected<'b', B>
const createNinja = (ctx: Injected<'ninjaWeapon', Weapon>) => ({
  slash(): number {
    switch (ctx.ninjaWeapon().kind) {
      case 'sword':
        return 1.1 * ctx.ninjaWeapon().damage;
      case 'firearm':
        return 0;
    }
  },
});





// every injected service is done lazily so that we avoid circular
// dependencies from the get go.
const container = createContainer()
  .bind('ninjaWeapon', Katana)
  .lazy('ninja', createNinja)
  .build();

const ninja = container.ninja();

console.log('Damage dealt:', ninja.slash());

const weapon = container.ninjaWeapon();
console.log('ninja weapon kind', weapon.kind);
```

### Validation

Some framework still relies on decorators for object validation,
one of the most popular library using decorators is class-validator, it's
the one mentioned in nest.js documentation.

class-validator has quite the history and is a proven solution for class
validation, but it still fall short in it's use of the typesystem; today
libraries like zod makes use of function composition and typescript
powerful type inference.

I will use it as an example of why you need to think carefully about
decorators as they do not play well with typescript.

```typescript
  ADMIN = 'admin',
  USER = 'user',

class Profile {
  @IsString()
  // Here we are making the mistake of having the value marked as optional
  // but it is not nullable in the type
  @IsOptional()
  @MinLength(3)
  bio!: string;

  @IsDate()
  @Type(() => Date)
  joinedDate!: Date;

  @IsOptional()
  @IsString()
  avatarUrl?: string;
}

class User {
  @IsString()
  @MinLength(2)
  name!: string;

  @IsEmail()
  email!: string;

  @IsEnum(UserRole)
  role!: UserRole;

  @ValidateNested()
  @Type(() => Profile)
  profile!: Profile;

  @Expose()
  @Transform(({ obj }) => `${obj.name} (${obj.role})`)
  displayName!: string;
}

const userData = {
  name: 'John Doe',
  email: 'john@example.com',
  role: 'admin' as UserRole,
  profile: {

    avatarUrl: 'https://example.com/avatar.jpg',

} satisfies Partial<User>;

// pass validation even if invalid, which is very unsafe for a
// validation library!
const user = plainToClass(User, userData);

validate(user).then((errors) => {

    console.log('Validation failed:', errors);
```

## Builder Pattern vs Fluent Function Chains

```typescript
class UriBuilder {
  private parts: string[] = [];
  add(part: string) {
    this.parts.push(part);
    return this;
  }
  build() {
    return '/' + this.parts.join('/');
  }
}

const product: string = new UriBuilder().add('hello').add('world').build();
console.log(product); // "/hello/world"
```
```typescript
const uriBuilder = (initial: string[] = []) => ({
  add: (part: string) => uriBuilder([...initial, part]),
  build: () => '/' + initial.join('/'),
});
// There are ways to get a string literal type at the end instead of string
const product: string = uriBuilder().add('hello').add('world').build();
```
